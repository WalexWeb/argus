# ARGUS — SIEM-lite Prototype

**АРГУС** — учебный прототип платформы для сбора, нормализации, корреляции и визуализации событий информационной безопасности.

> Текущий проект реализует основную цепочку SIEM: ingestion → normalization → deduplication → correlation → alerting → dashboard.

---

## 🚀 Обзор

Проект состоит из двух независимых приложений:

- `argus-server/` — backend на Nest.js с in-memory хранилищем и аналитикой.
- `argus-client/` — frontend на Next.js с дашбордом, таблицами, и корреляционной визуализацией.

Проект не является production-ready SIEM. Он демонстрирует архитектуру и логику, но хранит данные только в памяти и не использует внешние БД, очереди или безопасность уровня корпоративной SIEM.


## 🧠 Что реализовано

- REST API для приёма и получения событий.
- Нормализация логов JSON, syslog и CSV.
- Дедупликация событий по SHA-256 хешу.
- Корреляция событий на основе JSON-правил из `argus-server/data/correlation-rules.json`.
- Алгоритмы для обнаружения:
  - множественных пользователей с одного IP;
  - доступа email из разных стран;
  - сценариев компрометации аккаунта;
  - IDS-событий высокой серьезности.
- Визуализация информации на дашборде, странице событий, алертов и корреляционной сети.


## ✅ Соответствие SIEM-логике

Проект реализует следующие SIEM-процессы:

- **Ingestion** — приём событий через API;
- **Normalization** — унификация структуры событий;
- **Deduplication** — исключение повторов;
- **Correlation** — сопоставление событий по правилам;
- **Alerting** — генерация алертов;
- **Dashboards** — обзорная аналитика и визуализация.

### Ограничения текущей реализации

- Нет постоянного хранилища: данные теряются при перезапуске.
- Нет PostgreSQL/Redis/BullMQ и связанных production-компонентов.
- Нет аутентификации и RBAC.
- Нет масштабируемого pipeline ingestion.
- Нет интеграций с реальными TI-источниками.

## 🌐 Основные API

### `POST /api/v1/ingest/mock`
Загрузить и проанализировать `data/mock-logs.json`.

### `POST /api/v1/ingest/event`
Принять одно событие через JSON. Схема запроса описана в `argus-server/src/common/schemas/event.schema.ts`.

### `GET /api/v1/ingest/status`
Статус: количество событий и загруженных правил.

### `GET /api/v1/events`
Список событий. Поддерживаются query-параметры `source`, `event_type`, `limit`.

### `GET /api/v1/events/:id`
Детали события по идентификатору.

### `GET /api/v1/alerts`
Список алертов. Можно фильтровать `?severity=critical|high|medium|low`.

### `GET /api/v1/rules`
Правила корреляции.

### `GET /api/v1/summary`
Обзорная статистика для дашборда.

### `GET /api/v1/correlation-graph`
Данные для визуализации корреляционной сети.



## 🧩 Ключевая архитектура

- **EventsService** — хранение событий в массиве памяти, базовые агрегации, фильтрация.
- **IngestionService** — парсинг raw-логов и нормализация структур.
- **CorrelationService** — загрузка правил и построение алертов.
- **DashboardService** — формирование summary и данных для графа.
- **Next.js UI** — рендер dashboard, таблицы событий, алертов и correlation view.


## 💡 Итог

Проект — хороший прототип SIEM-системы с рабочим потоком событий и корреляцией. Для полноценной SIEM ему не хватает долговременного хранилища, безопасности и интеграций, но архитектура уже отражает основные этапы обработки событий.
